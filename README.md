# BimariHaunter
> **Stay Informed. Stay Safe.**
BimariHaunter is a real-time public health Android application built for Pakistan. It aggregates outbreak and disease reports from across the web, surfaces them on an interactive map, and delivers proximity-aware notifications so users are alerted the moment a health threat enters their area.

## Overview

Pakistan faces recurring outbreaks of dengue, cholera, typhoid, and other preventable diseases — yet most citizens have no reliable, real-time way to know what is happening near them. BimariHaunter solves this by combining a cloud-based RSS scraper, an AI classification pipeline, and a mobile-first interface that puts live outbreak intelligence directly in the user's pocket.

The app is designed for three audiences: **individuals** who want to know if there is a health risk near their home or travel route; **community groups** who want to discuss and share local health information; and **researchers** who want a quick visual overview of disease spread patterns across the country.

## Features

### Live Disease Feed
The feed reads directly from the `/reports` Firestore collection, which is populated continuously by the cloud scraper backend. Each report is cached locally in a Room database so the feed remains usable offline. Reports can be filtered by disease type, severity, and location. The feed supports pull-to-refresh and displays the source, publication timestamp, AI-assigned severity, and a plain-language summary generated by the classification model.

### Interactive Disease Map
The map renders outbreak pins from the local Room cache. Pins are loaded dynamically as the user pans and zooms — only markers within the current camera viewport are rendered, keeping performance smooth even with hundreds of reports. Each marker snippet shows the disease name and a human-readable "Updated X minutes ago" timestamp derived from `scraped_at`. A refresh button in the bottom panel triggers an immediate Firestore sync.

### 15km Proximity Alerts
When the user's GPS position comes within 15 kilometres of any outbreak pin, BimariHaunter fires both an in-app ghost banner (with severity colour coding) and a system notification. The same check runs passively as the user travels, so the app functions as a silent sentinel during commutes and journeys.

### Live Insights
The Insights tab computes analytics in real-time from the cached reports: a "Today's Story" card highlighting the most critical active outbreak, a disease breakdown bar chart, a severity distribution chart, and a reports-per-day trend line. All charts are drawn from live data — there are no hardcoded values.

### Community Chat
Users can join public group chats organised around specific outbreaks or regions, send direct messages to friends, and create new groups. The chat system is backed by Firestore subcollections and supports real-time message delivery.

### Friend System with QR Codes
Users can search for friends by UID or scan a friend's QR code directly from the camera. Each user profile displays a unique QR code containing their UID. The friend request flow uses Firestore to store pending and accepted connections.

### Offline AI Assistant
The app bundles a MediaPipe LLM Inference session running a quantised Llama 3.2 1B model on-device. When the user has no internet connection, the AI assistant answers health queries using only the locally cached outbreak reports as context, ensuring it never fabricates information it does not have.

### Push Notifications
Firebase Cloud Messaging delivers four notification types: outbreak alerts (with proximity filtering on the server side), direct messages, friend requests, and location-change alerts when the user moves to a new city.

## System Architecture

![System Architecture](architecture.png)

BimariHaunter follows a **unidirectional data flow** architecture with a clear separation between the data layer, domain layer, and presentation layer.

The backend scraper runs on Google Cloud Run and writes classified reports to Firestore. The Android app reads those reports via a Firestore real-time listener, caches them in Room, and serves all UI from the local cache. This means the UI is always fast and the app degrades gracefully when offline.

---

## Data Flow

![Data Flow](dataflow.png)

The sequence above shows the four key flows in the system:

1. **Continuous Scrape Cycle** — The backend polls RSS feeds, classifies each article, geocodes the locations, and writes a structured document to `/reports`.
2. **Real-Time Sync** — The Firestore snapshot listener on the Android app fires immediately when a new document arrives, maps it to a Room entity, and the UI updates automatically.
3. **Proximity Alert Flow** — On every GPS update, the app runs a Haversine distance check against every cached report. If the user is within 15km, a ghost banner and system notification fire.
4. **Manual Refresh** — The user can tap the refresh button to trigger an immediate scrape cycle via the backend API.

## Data Model

### Firestore `/reports/{docId}`

| Field | Type | Description |
|-------|------|-------------|
| `title` | String | Headline of the report |
| `source` | String | Publishing outlet name |
| `url` | String | Original article URL |
| `raw_text` | String | Full scraped text |
| `published_at` | Timestamp | Original publication time |
| `scraped_at` | Timestamp | When the scraper collected this |
| `disease` | String | AI-classified disease type |
| `severity` | String | `low` / `medium` / `high` |
| `ai_analysis.summary` | Array\<String\> | Plain-language summary sentences |
| `ai_analysis.locations` | Array\<String\> | Detected location names |
| `ai_analysis.coordinates` | GeoPoint | Lat/lon for map pin |
| `ai_analysis.confidence_score` | Double | Model confidence (0–1) |
| `content_hash` | String | SHA-256 for deduplication |
| `status` | String | `analyzed` / `pending` |

### Room `outbreak_reports`

The Room entity mirrors the Firestore document with flattened `ai_analysis` fields. `List<String>` fields are stored as JSON via `OutbreakConverters`. The database version is 3 with `fallbackToDestructiveMigration` enabled — the cache is always rebuildable from Firestore.

### Firestore `/users/{uid}`

Stores display name, email, profile photo URL, location, notification preferences, and a `friends` subcollection. User documents are readable only by authenticated users and writable only by the document owner.

## Setup & Installation

### Prerequisites

- Android Studio Hedgehog (2023.1.1) or later
- Android SDK 34
- A Firebase project with Firestore, Auth, and FCM enabled
- Google Maps API key

### Steps

1. Clone the repository and open it in Android Studio.
2. Place your `google-services.json` in `app/`.
3. Add your Google Maps API key to `local.properties`:
   ```
   MAPS_API_KEY=your_key_here
   ```
4. (Optional) Place a compatible MediaPipe `.task` model file at `app/src/main/assets/models/llama-3.2-1b-instruct.task` to enable the offline AI assistant.
5. Sync Gradle and run on a device or emulator with Google Play Services.

## Firebase Configuration

### Firestore Security Rules

Apply the following rules in the Firebase console under **Firestore → Rules**:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    match /reports/{doc} {
      allow read: if true;
      allow write: if false;
    }

    match /users/{userId} {
      allow read: if request.auth != null;
      allow write: if request.auth != null && request.auth.uid == userId;
      match /friends/{doc}   { allow read, write: if request.auth != null && request.auth.uid == userId; }
      match /blocked/{doc}   { allow read, write: if request.auth != null && request.auth.uid == userId; }
    }

    match /chats/{chatId} {
      allow read: if true;
      allow write: if request.auth != null;
      match /messages/{msgId} { allow read: if true; allow write: if request.auth != null; }
    }

    match /groups/{doc}           { allow read: if true; allow write: if request.auth != null; }
    match /direct_chats/{chatId}  { allow read, write: if request.auth != null;
      match /messages/{msgId}     { allow read, write: if request.auth != null; } }
    match /friend_requests/{doc}  { allow read, write: if request.auth != null; }
    match /scrape_jobs/{doc}      { allow read: if request.auth != null; allow write: if false; }
  }
}
```

> **Important:** The default rules shipped with new Firestore projects are `allow read, write: if false`. This blocks everything. The rules above must be published before the app can read any data.

## Backend

The backend is a Python service deployed on Google Cloud Run at:

```
https://bimari-haunter-723264184490.asia-southeast1.run.app/
```

It exposes a `POST /trigger-job` endpoint that the Android app calls (with a Firebase ID token) to request an immediate scrape cycle. The scraper fetches RSS feeds from Pakistani and international health news sources, runs each article through a keyword classifier (`rss-keyword-classifier-v2`) to detect disease mentions and severity, extracts coordinates via a geocoding step, and writes the result to `/reports` in Firestore.

The Android app authenticates all backend requests through `AuthInterceptor`, which calls `FirebaseAuth.currentUser.getIdToken(true)` on a background thread and injects the resulting JWT as a `Bearer` token.

## Contributing

This project was built for a hackathon. If you want to extend it, the most impactful areas are:

- **Expanding the scraper** to cover more RSS sources and languages (Urdu support is a priority).
- **Improving the AI classifier** — the current `rss-keyword-classifier-v2` is keyword-based; a fine-tuned transformer would significantly improve precision.
- **Adding heatmap layers** to the map for visualising outbreak density.
- **Implementing the Alerts tab** with user-configurable disease and location filters.
- **Internationalisation** — the UI is English-only; Urdu RTL support would dramatically expand the user base.
